內容品質問題：一週 6/7 天全 noise
======================================================================

問題回報
----------------------------------------------------------------
使用者反映「本週記錄 冷 的狀況是否太多了」。統計近一週 history.json，
6/27~7/3 共 7 天，只有 7/1 有 core 條目，其餘 6 天全部 noise。

第一輪根因與修正
----------------------------------------------------------------
根因：fetch_rss()/fetch_ddg() 只抓標題+200字內截斷摘要，從未抓取
文章完整內文；LLM 材料不足時只能套用空洞模板，被下游品質閘正確攔截
為 noise，但攔截頻率過高。

修正：新增 fetch_article_text(url)（regex 擷取 <p> 段落，跳過
nav/footer/script）+ enrich_with_full_text(snippets, max_fetch=8)：
對高信號候選新聞抓取真實文章內文（最初 500 字）取代薄摘要，enrich
成功的條目移到陣列最前面，確保 news_short[:3500] 截斷時優先保留。
scripts/update_data.py 與根目錄 update_data.py 同步套用（根目錄原本
缺 HIGH_SIGNAL_PAT，一併補上）。

同時發現另一個獨立問題：isNoiseEmpty(item) 判斷式要求 body 含
「無相關新聞」字樣才隱藏，但 make_empty_day() 早已改回傳空陣列，
該字串已無處產生，導致後端正確判為 noise 的套模板重複內容仍完整
顯示（7/3 五張卡片皆為同一模板換詞）。改為只看 rating==='noise'。

第二輪：whack-a-mole 教訓
----------------------------------------------------------------
連續兩輪 rerun 後，每次都被同一道品質閘正確攔下，但每次「表面症狀」
的具體措辭都不同（提高服務品質/創造就業機會 → 成長率預測）。逐字加
FORBIDDEN_PATS 對這類情況沒有用——內容本來就會被既有 bigram 重複偵測
正確攔下，加 pattern 不會改變任何結果，純粹浪費時間。

使用者明確指出「你浪費太多時間了，請重新確認root cause後修復」。
真正 root cause：prompt schema 寫死「EXACTLY 3 sentences」，逼 LLM
只有 1 個事實也要湊滿 3 句，湊不出來就套用固定的幾種模板。

結構性修正（停止繼續加 pattern）：
1. body 句數改為「1-3 句，視實際掌握事實數量而定」，明講湊句數
   沒有任何好處（反正會被判 noise）
2. noise 佔位條目規則明確化：素材不足時必須老實寫「今日無相關新聞」，
   禁止捏造「AI應用進步」這類空泛假標題
3. scripts/update_data.py 的 call_groq() system message 補上
   noise 鐵律（根目錄版本早有此規則但兩檔案不同步）

f-string 崩潰事故
----------------------------------------------------------------
上述修正 push 後，使用者 rerun 時 GitHub Actions 在 make_prompt()
直接 NameError: name 'section' is not defined 崩潰——RULES 文字裡
誤寫了 {section}，被 f-string 當成變數插值，但函式作用域內沒有這個
變數。python3 -m py_compile 完全抓不到這種錯誤（語法合法，只有
runtime 才會拋例外）。改為靜態文字「該分類」，並改用「實際呼叫
make_prompt() 執行一次」而非只做語法檢查來驗證。
崩潰點在 call_groq() 之前，故無 API 費用浪費，data/history.json
也沒被寫入或污染。

第三輪：真正的 root cause——數字偵測 regex 漏中文數字
----------------------------------------------------------------
使用者質疑「這個要跟你確認過去做過的修正是不是都有問題」+
「你不是應該要讀記憶裡的spec才動手嗎」——事後補讀 spec 才確認
debt 公式規則。

實際比對這輪 rerun 內容，找到「Google 計劃在南非建立連接中心」——
body 完全真實、有效來源、無空話無重複，卻仍被判 noise。追查
_body_is_low_quality() 判斷「是否含具體數字」的 regex 只認阿拉伯
數字（\d）與幾個大數單位（億/兆/倍/萬），完全沒涵蓋中文數字。這篇
的關鍵事實「建立四個連接中心」「第一個中心」用中文數字表達，regex
認不出來，被誤判「無具體數字」而降評 noise——跟內容品質無關，是
偵測邏輯本身的盲點，屬於偽陽性，與先前「內容真的空洞被正確攔下」
是完全不同類型的問題。

修正：regex 加入中文數字+量詞組合
（[一二三四五六七八九十兩]+[個家款次項座台支波批輪席人年月日]、
第[一~十]）。比對近一週歷史資料，同類誤殺至少發生過 2 次。

第四輪：SOURCE_URL 位置問題（v3.1 修正自己造成的副作用）
----------------------------------------------------------------
數字 regex 修正後 rerun，驗證「Anthropic」「Google」兩篇都通過了
low_quality 與 cross-item 重複檢查，body 也真實無套話——但兩者
source 欄位都是空字串，觸發 downgrade_unsourced 判定「core/opp
但無來源」而降評 noise。

根因：enrich_with_full_text() 把格式寫成
「標題 — [1100字內文]SOURCE_URL:xxx」，內文拉長後 SOURCE_URL 被埋
在整段文字最後面，LLM 常常讀到文末前就已經寫完該筆 item 的 JSON，
導致 source 留空或抓錯（前一輪 Qolab 案例就是誤抓成另一篇文章的
URL，同一成因）。

修正：格式改為「標題SOURCE_URL:xxx — [內文]」，網址緊接標題後面、
在長內文之前；prompt 加明確提醒。

第五輪：insight 欄位漏檢查
----------------------------------------------------------------
使用者指出某輪 email report 裡「注意方向」（insight 欄位）用「並且
可能會對其競爭對手的X業務產生負面影響」這句在 Nvidia/Google/
SoftBank/Qolab 四篇完全不相關新聞裡逐字重複——追查
downgrade_forbidden_phrases() 只檢查 body 和 impact，insight 欄位
從建立以來就沒被這道檢查覆蓋到，等於是現成的漏洞。使用者當下明確
要求「不要修東西又影響到html呈現」，這次修正只動了兩個
update_data.py，完全沒碰 ai_tracker_v6.html。

修正：downgrade_forbidden_phrases() 新增 insight 檢查分支；
FORBIDDEN_PATS 新增「對其競爭對手的...產生負面影響」等套話 pattern。

使用者釐清真實需求後，發現 v3.2 顯示修正方向是錯的
----------------------------------------------------------------
使用者講出三個真實需求：①內容品質 ②Gmail 推送內容應該等於 html
「本週紀錄」內容 ③過去已研究/做過筆記的歷史新聞現在看不到了要修正。

回頭檢視稍早的 isNoiseEmpty 修正（rating==='noise' 就整張隱藏），
發現這個規則套用在全部 7 天歷史紀錄上，不只是今天——回測 6/27~7/2
每天有 2~5 筆被標 noise 但內容其實是真實新聞的條目，全部被連帶
藏起來。email 範本從不篩選 rating，html 隱藏 noise 之後兩邊呈現
從此不一致。

最終修正：isNoiseEmpty 改回「只隱藏真的沒有 body 內容的空白佔位」，
不再依 rating 隱藏；newsCard() 新增灰底 NOISE 徽章，noise 內容照常
顯示，只是視覺上標示品質等級——與 email 呈現邏輯一致，過去的歷史
資料也全部恢復可見。附帶修好「個人洞察」分頁的 NOISE 子分頁（本來
就設計好但一直因為同一個過濾規則永遠是空的）。

本次會話重要教訓（已寫入記憶檔 feedback_no_whack_a_mole_debugging.md
與 feedback_confirm_filter_scope.md）
----------------------------------------------------------------
1. 同一症狀類別連續多輪用不同措辭出現時，停止逐字堵漏洞，要往上一層
   找生成端的結構性強制要求，不要對症狀加 regex。
2. 修改「顯示/隱藏什麼」的邏輯前，先確認影響範圍是否回溯波及歷史
   資料，以及是否需要與其他輸出管道（email）保持一致。
3. f-string 內容修改後，僅 py_compile 語法檢查不夠，須實際執行函式
   驗證 runtime 錯誤。
4. 讀 spec 才是規則的權威來源，記憶檔只給架構輪廓；涉及「公式/規則
   對不對」的問題應優先讀 spec，不要自己用資料反推規則。
